# Архитектурный манифест The Fool's Reef (v6.2)

Этот документ является «конституцией» проекта. Любой новый код должен строго соответствовать изложенным принципам для поддержания изоляции логики и масштабируемости системы.

## 1. Фундаментальные принципы
*   **Изоляция**: `GameController` (логика) не знает о существовании `GuiActor` и `pygame`. `Activity` (визуал) не знает правил игры.
*   **Плоская структура**: Управление жизненным циклом экранного представления сосредоточено в активном `GameScreen`. Никаких лишних менеджеров-посредников для оркестрации.
*   **Экран не является бог-объектом**: `GameScreen` владеет экранными зонами, активным набором actor ID, маршрутизирует ввод и запускает визуальные процессы, но не хранит правила игры и не содержит сложные сценарии анимации. Сложная временная визуальная логика выносится в `Activity`.
*   **Единый склад акторов**: Все долгоживущие `GuiActor` создаются при старте игры в отдельном контейнере графических объектов. Экраны не создают и не хранят все акторы игры, а берут нужные акторы из этого контейнера по ID.
*   **Разделение зон**: `GameController` владеет логическими зонами игры, а `GameScreen` владеет экранной геометрией этих зон. Одна и та же зона имеет две проекции: смысловую и визуальную.
*   **Двунаправленность**:
    *   *Input (Visual → Logic)*: События ввода обрабатываются экраном, транслируются в ID объектов и передаются в `GameController`.
    *   *Output (Logic → Visual)*: Логика инициирует визуальные процессы, которые «хореографируют» анимацию.
*   **Принцип инъекции ресурсов**: Ни один объект (кроме `GameScreen` на этапе сборки) не имеет права запрашивать ресурсы. Все визуальные данные передаются в объекты при создании.
*   **Самодокументированные ресурсы**: Параметры спрайт-листов (размеры кадров, количество) хранятся непосредственно в метаданных PNG-файлов.

## 2. Роли элементов архитектуры

### A. RenderEngine (`/core/render_engine.py`)
*   **Функция**: Системный цикл Pygame, управление FPS, вызов `update/draw`.
*   **Ограничение**: Никакой игровой логики.

### B. GameController (`/core/game_controller.py`)
*   **Функция**: Хранилище правил и состояния игры.
*   **Логические зоны**: Владеет составом игровых зон: рука игрока, стол, колода, сброс и другие зоны правил.
*   **Взаимодействие**: Принимает решения о ходе игры и дает визуальные «распоряжения» через `GameScreen` (например: «запустить активность X для объекта Y»).
*   **Ограничение**: Никакого визуального кода. Работает только с абстрактными данными.

### C. GameScreen (`/screens/*.py`)
*   **Множественность**: В игре может существовать несколько экранов. Каждый конкретный `GameScreen` описывает одну визуальную сцену: главный стол, меню, экран настроек, экран результатов и так далее.
*   **Владение**: Хранит активный набор actor ID для своего экрана и список активных `Activity`.
*   **Ввод**: Фильтрует клики по координатам через `hit_rect` акторов, определяет ID актора и транслирует в `GameController.on_actor_clicked(id)`.
*   **Доступ к акторам**: Получает нужные `GuiActor` из общего контейнера графических объектов по ID. Не является складом всех графических объектов игры.
*   **Экранные зоны**: Владеет геометрией зон конкретного экрана: прямоугольниками, точками привязки, правилами размещения акторов внутри зоны и порядком их отрисовки.
*   **Оркестрация**: По заданиям `GameController` выбирает нужные акторы, размещает их в своих экранных зонах, обновляет активности и активные акторы в `update(dt)`.
*   **Граница ответственности**: Не реализует правила игры и не превращает анимационные сценарии в длинные блоки кода внутри экрана. Если визуальный процесс имеет длительность, этапы или завершение, он оформляется как `Activity`.

### D. GuiActorStore (`/actors/gui_actor_store.py`)
*   **Функция**: Единый контейнер созданных графических объектов игры.
*   **Инициализация**: При старте игры получает ресурсы от `ResourceManager`, создает все базовые `GuiActor` и сохраняет их по стабильным ID.
*   **Доступ**: Предоставляет `GameScreen` доступ к `GuiActor` по ID.
*   **Ограничение**: Не содержит правил игры, не обрабатывает ввод, не запускает активности и не принимает решений о размещении акторов на экране.

### E. Activity (`/activities/*.py`)
*   **Функция**: Временный визуальный «хореограф».
*   **Принцип**: Получает ссылки на существующие `GuiActor` и управляет их визуальным состоянием (позиция, прозрачность, активный кадр) в течение своего жизненного цикла.
*   **Жизненный цикл**: Имеет состояние выполнения и сообщает `GameScreen`, когда завершена.
*   **Ограничение**: Полное отсутствие логики игры. Не меняет состояние `GameController` напрямую.
*   **Завершение**: После завершения `Activity` экран удаляет ее из списка. Если после визуального процесса нужно продолжить игровой сценарий, решение принимает `GameScreen` совместно с `GameController`, а не сама `Activity`.

### F. GuiActor (`/actors/gui_actor.py`)
*   **Функция**: Пассивный «холст». Хранит базовый `rect`, обязательный `hit_rect`, графические слои и их параметры (`Surface`, `position`, `alpha`, `current_frame`, `visible`).
*   **Базовая геометрия**: `rect` задает положение актора на экране. `hit_rect` задает область взаимодействия и может отличаться от визуального размера.
*   **Слои**: Каждый слой имеет имя, тип данных, локальное смещение относительно `rect`, прозрачность, видимость и текущий кадр, если слой анимированный.
*   **API слоя**: `GuiActor` обязан предоставлять методы для изменения визуальных свойств слоя: позиция, смещение, прозрачность, видимость, текущий кадр.
*   **Принцип**: Предоставляет API для изменения визуального состояния. Не имеет логики анимации и не принимает игровые решения.

### G. ResourceManager (`/core/resource.py`)
*   **Функция**: Единственный источник «сырых» графических данных. Загружает, кэширует и отдает готовые `Surface`.
*   **Принцип**: Считывает метаданные (размер кадра, кол-во) прямо из тела PNG-файла.

## 3. Схема взаимодействия (Data Flow)

### Zone Flow (Logic Zone → Screen Zone)
1. **Логическое состояние**: `GameController` хранит, что `card_id` находится в логической зоне, например `bottom_hand`.
2. **Визуальная интерпретация**: `GameScreen` знает, где зона `bottom_hand` расположена на экране и как разложить в ней акторы.
3. **Получение представления**: `GameScreen` берет `GuiActor` из `GuiActorStore` по тому же `card_id`.
4. **Активация на экране**: `GameScreen` добавляет `card_id` в активный набор actor ID своего экрана.
5. **Размещение**: `GameScreen` рассчитывает `rect`, `hit_rect` и локальные параметры слоев актора.
6. **Визуальный процесс**: Если появление, перемещение или удаление должно быть анимировано, экран запускает соответствующую `Activity`.

### Input Flow (Visual → Logic)
1. **Событие**: Пользователь кликает ЛКМ по `GuiActor`.
2. **Детекция**: `GameScreen` проверяет `hit_rect` акторов и находит ID актора.
3. **Трансляция**: `GameScreen` вызывает `GameController.on_actor_clicked(id)`.
4. **Решение**: `GameController` принимает решение согласно правилам игры.

### Output Flow (Logic → Visual)
1. **Команда**: `GameController` дает команду `GameScreen`: *"Запустить активность типа X для объекта Y"*.
2. **Выбор актора**: `GameScreen` получает нужный `GuiActor` из `GuiActorStore` по ID.
3. **Инициализация**: `GameScreen` создает экземпляр `Activity` и добавляет его в список `active_activities`.
4. **Визуализация**: В цикле `update()` `GameScreen` вызывает `activity.update(dt)`. Активность меняет свойства `GuiActor`.
5. **Финал**: По завершении работы `Activity` сообщает о завершении, а `GameScreen` удаляет ее из списка.
6. **Продолжение сценария**: Если после визуализации нужно изменить ход игры, `GameScreen` обращается к `GameController`. `Activity` не меняет игровое состояние напрямую.

## 4. Золотые правила разработки
1. **Логика не касается графики**: Если нужно изменить механику — правьте `GameController`.
2. **Графика не касается правил**: Если нужно изменить анимацию — правьте `Activity`.
3. **Инъекция ресурсов**: Все данные передаются в объекты при создании. Никаких импортов `ResourceManager` внутри `GuiActor` или `Activity`.
4. **Единый экранный оркестратор**: Активный `GameScreen` — единственный авторитет в вопросах размещения акторов на своем экране и жизненного цикла визуальных процессов.
5. **Абстракция ввода**: Контроллер оперирует ID сущностей, экран — координатами пикселей.
6. **Самодокументированность**: Метаданные нарезки спрайтов всегда находятся внутри самого PNG-файла.
7. **Обязательный hit_rect**: Каждый `GuiActor` имеет `hit_rect` как базовую область взаимодействия.
8. **Слойный API вместо скрытой магии**: Любые изменения визуального состояния проходят через явные методы `GuiActor`, а не через прямое изменение внутренних словарей.
9. **Activity завершается, но не решает правила**: Визуальный процесс может сообщить о завершении, но не должен сам менять состояние игры.
10. **Логическая зона не равна экранной зоне**: `GameController` хранит состав зоны, `GameScreen` хранит ее координаты и правила размещения.
11. **Экран не является складом акторов**: `GameScreen` хранит только активный набор actor ID, а сами `GuiActor` живут в `GuiActorStore`.
